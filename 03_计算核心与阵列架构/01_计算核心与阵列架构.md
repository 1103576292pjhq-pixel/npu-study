# 03 计算核心与阵列架构

前两卷解决了“为什么需要 NPU”和“看 NPU 应该用什么坐标轴”。这一卷开始进入真正的计算核心：`谁在做乘加`、`谁在保存部分和`、`谁在搬数据`、`谁在决定执行节奏`。如果这一层没看清，后面的数据流、量化、编译优化都会失去着力点。

## 1. 先把问题缩到最小：一个输出值是怎么被算出来的

无论是卷积、GEMM 还是 attention 内部的大块矩阵乘，核心动作都可以拆成：

1. 取一组输入激活
2. 取一组权重
3. 做乘法
4. 把结果累加到部分和
5. 在归约结束后写回输出

围绕这五步，计算核心要解决三件事：

- 乘法和加法由哪个最小单元执行
- 部分和在什么位置累积
- 激活、权重、部分和怎样在阵列与存储之间流动

所以看任何 NPU 计算核心，第一视角都应该是`归约机器`，不是“大模块拼盘”。

## 2. MAC 是原子，PE 是工程上的最小执行单元

### 2.1 MAC：Multiply-Accumulate

MAC 的逻辑很直接：

`psum = psum + a * b`

但真正做硬件时，MAC 并不只是一个乘法器加一个加法器，还要同时处理：

- 输入位宽
- 累加位宽
- pipeline 深度
- 饱和 / 截断 / 舍入策略
- 时钟频率与面积

尤其在低精度推理里，乘法位宽可能是 `INT8 x INT8`，但累加路径通常需要更高位宽，否则部分和会很快溢出。

### 2.2 PE：Processing Element

PE 是把 MAC 放进可复用执行环境后的工程单元。一个最小 PE 通常至少包含：

- 乘加逻辑
- 本地寄存器或小缓存
- 输入/输出通路
- 简单控制状态

它回答的问题是：`一个单元在一个节拍或一小段时间里，能稳定完成什么计算并与邻居交换什么数据`

所以 MAC 是原子运算，PE 是原子执行单元。两者不要混用。

## 3. 计算核心通常不是单一单元，而是分层组合

从本地课程里抽出来看，很多 NPU IP 都会把核心拆成几类部件：

- `SPU / Scalar Unit`
  - 负责轻量控制、标量运算、配置、地址或辅助逻辑
- `VPU / Vector Unit`
  - 负责规则向量操作，适合元素级并行、layout 转换、部分预后处理
- `TPU / Tensor Unit`
  - 负责矩阵乘加主吞吐，通常是阵列里最“吃硅”的部分

这种拆法的本质是把不同粒度的任务分层：

- 标量控制不应该占用大阵列
- 大阵列不应该被迫去执行零碎控制
- 向量单元可以填补“不是纯 GEMM、但又足够规则”的中间地带

因此一台 NPU 往往不是只有一个“巨大阵列”，而是`标量 + 向量 + 张量`三级配合。

## 4. 脉动阵列为什么会反复出现

脉动阵列不是流行词，而是解决数据复用问题的直接结构答案。

它的核心思想很简单：

- 把大量 PE 排成二维或更高维阵列
- 让权重、激活、部分和按固定方向在阵列中流动
- 每个 PE 在数据经过自己时做局部乘加
- 数据不必每次都回到全局存储再取一次

这种结构的优势：

- 控制规则，节奏清楚
- 局部数据复用强
- 容易把计算和搬运重叠
- 对 GEMM/卷积这类规则计算非常友好

它的代价：

- 对不规则形状和稀疏模式不够灵活
- 阵列维度、tile 大小、边界填充都会影响利用率
- 编译器必须会“喂阵列”，否则大量 PE 会空转

所以脉动阵列不是“最佳架构”的同义词，而是对规则乘加负载非常有效的一种空间组织。

## 5. 看阵列，不只看有多少 PE，要看三种数据谁在动

一个阵列结构到底高不高效，不是看 PE 数量，而是看三种数据的流动关系：

- `权重`
- `激活`
- `部分和`

这三者决定了经典的数据流分类：

- `Weight Stationary`
  - 权重尽量不动，适合权重复用强的场景
- `Output Stationary`
  - 部分和留在本地直到归约结束，减少回写
- `Input/Activation Stationary`
  - 激活尽量留本地，适合特定访问模式

真正的阵列设计不是抽象地“做并行”，而是在回答：

- 哪类数据最贵
- 哪类数据最值得留在本地
- 哪类数据更适合流动

后面讲 dataflow 时会专门展开，但在计算核心层就必须先形成这个意识：`阵列结构 = 计算单元组织 + 数据驻留策略`

## 6. 向量单元和张量单元的分工

很多初学者会把“向量并行”和“张量阵列”混成一件事。两者既相关，又不相同。

### 6.1 向量单元

向量单元更像把同一种标量操作扩展到一整条 lane 上：

- 元素加减乘
- 比较、选择、激活
- layout 转换
- 向量 load/store

它的优点是灵活、通用、适合规则元素级操作。

### 6.2 张量单元

张量单元更强调：

- 批量乘加
- 归约
- 局部复用
- 与阵列数据流强耦合

它的优点是对 GEMM/卷积吞吐极高，但灵活性通常弱于向量单元。

所以实际 NPU 里常见的模式是：

- 张量单元负责主计算洪峰
- 向量单元负责边界处理、轻量算子、变换和辅助操作
- 标量单元负责配置与控制

## 7. 计算核心不是只有算，还包含数据路径

很多“架构图”只画阵列方块，但真正决定效率的往往是数据路径：

- 激活从哪里进入
- 权重从哪里进入
- 部分和在哪里汇合
- 是否支持广播、多播、级联
- 阵列与本地 buffer 之间是否能并行读写

一个高效计算核心，至少要让下面几件事同时成立：

1. `供数不断`
   - 阵列不因等待数据而空转。
2. `归约不断`
   - 部分和不会频繁被迫回写再读回。
3. `模式可切`
   - 不同层形状下仍能维持较高利用率。
4. `控制不过载`
   - 不用把太多能量花在复杂调度和细碎同步上。

所以“数据通路设计”不是附属模块，它和 MAC 阵列一样属于计算核心本体。

## 8. 精度支持会反过来重塑核心结构

支持不同精度，不是多挂几个模式位那么简单。它会直接改动：

- 单个 MAC 的面积
- 同一阵列里能并排放多少 lane
- 累加树深度和位宽
- 寄存器文件和 buffer 宽度
- 数据搬运压力

典型现象：

- 同样面积下，`INT8` 阵列往往能放下更多并行 lane
- 低精度能减轻存储和带宽压力
- 但混合精度或多精度支持会让控制和格式转换复杂化

所以“支持 FP16/INT8/INT4”应该被看成架构选择，不只是功能列表。

## 9. 一个计算核心好不好，要看什么

判断一套核心设计，不妨按下面五个问题逐项看：

1. `归约路径是否清晰`
   - 部分和在哪里累积，何时回写，是否容易溢出或抖动。
2. `数据复用是否真实存在`
   - 权重、激活、部分和是不是在高成本层级间反复搬动。
3. `阵列利用率是否容易做高`
   - 边界 tile、小 batch、不规则 shape 会不会把阵列打空。
4. `精度与位宽设计是否协调`
   - 乘法宽度、累加宽度、转换开销是否匹配目标负载。
5. `软件是否喂得动`
   - 编译器和 runtime 是否能把真实模型映射到这种执行组织上。

这五条里，前四条偏硬件，第五条决定前四条能不能落地。

把它压成读图顺序，就是：

1. 先找张量主计算路径，确认谁在做乘加。
2. 再找部分和停在哪里，确认归约路径。
3. 再看权重和激活从哪里进、往哪里走。
4. 最后看控制与精度切换是否会把前面三步拖慢。

## 10. 常见计算核心形态

| 形态 | 强项 | 常见问题 |
| --- | --- | --- |
| 纯向量型 | 灵活、易覆盖多种算子 | 对大规模归约吞吐不够极致 |
| 脉动阵列型 | 规则乘加效率高、复用好 | 对不规则工作负载不够友好 |
| 向量 + 张量混合型 | 在吞吐和灵活性之间折中 | 控制更复杂，软件映射更难 |
| 多核阵列型 | 易扩展峰值算力 | 互联、同步、片上存储压力上升 |

这也是为什么不同厂商的 NPU 看上去差异很大，但核心选择总绕不开：`阵列强度、向量能力、存储层级、控制复杂度` 这几根杠杆。

## 11. 学完本章后应形成的最小图景

至少应该能把一套 NPU 计算核心还原成下面这张抽象图：

`控制/标量单元 -> 向量单元 -> 张量阵列 -> 本地累加/缓存 -> 共享buffer/内存接口`

再进一步，能够回答：

- MAC、PE、阵列三者是什么关系。
- 为什么部分和的位置经常比乘法器数量更关键。
- 为什么数据流和阵列组织其实是一体两面。
- 为什么向量单元、张量单元和标量单元往往要同时存在。
- 为什么精度支持会影响整个核心结构，而不是局部小功能。

## 12. 常见误区

- 误区：`PE 越多，性能一定越高`
  - 修正：没有供数、归约和映射支撑时，更多 PE 只会制造更多空转面积。
- 误区：`脉动阵列就是 NPU 的唯一正确答案`
  - 修正：它非常适合规则乘加，但并不自动覆盖所有工作负载。
- 误区：`向量单元只是附属配角`
  - 修正：很多边界处理、元素级算子、layout 变换都离不开它。
- 误区：`多精度支持只是营销参数`
  - 修正：位宽设计会直接重塑阵列规模、存储宽度和能效上限。

这一卷的任务不是穷举所有 NPU 架构，而是建立一个足够强的抽象：以后看到任意平台，只要把它拆回 `MAC / PE / 阵列 / 数据路径 / 精度支持 / 控制方式` 六个维度，就不会被命名风格带偏。

下一卷会把这个抽象继续往下压到数据流、硬件循环和片上存储层级，真正解释“为什么同样的阵列规模，效率会差这么多”。
