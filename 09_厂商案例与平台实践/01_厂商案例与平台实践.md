# 09 厂商案例与平台实践

前面八卷已经建立了五条核心因果链：

1. `工作负载 -> 为什么需要 NPU`
2. `计算核心 -> 数据流 -> 吞吐`
3. `低精度 / 稀疏 / 融合 -> 更高有效计算密度`
4. `图语义 -> 编译计划 -> runtime 执行`
5. `性能 / 功耗 / 安全 / 多核 / Chiplet -> 系统上限`

进入厂商案例后，最容易重新掉回“记平台名”和“记产品参数”的陷阱。正确方式不是背型号，而是问：`这家平台把系统预算押在了哪里`

## 1. 先统一比较框架

看任何 NPU 平台，至少用同一张表去拆：

| 维度 | 要问的问题 |
| --- | --- |
| `目标场景` | 是 MCU、手机、边缘盒子、云端，还是异构 SoC |
| `计算核心` | 是偏阵列、偏向量、还是混合组织 |
| `数据流与存储` | 它更依赖哪些复用和片上缓存策略 |
| `软件栈` | 编译器、runtime、插件、SDK 是否成熟 |
| `系统取舍` | 它把预算主要押在能效、通用性、开发便利还是扩展性上 |

这五个问题比“TOPS 有多少”更能解释平台差异。

## 2. 边缘超低功耗路线：Arm Ethos-U

从本地课件里能看出，Ethos-U 系列的核心关键词是：

- 面向边缘和 MCU / 小型 SoC 场景
- 明确强调低功耗部署
- 通过较规整的计算阵列、权重缓存、激活缓存来支撑推理
- 编程模型与软件栈紧密绑定

Arm 官方产品 brief 把它说得更具体：Ethos-U 不是泛化的“大 NPU”，而是放在 `Cortex-M + microNPU` 的 endpoint AI 路线里，软件侧还明确配 `Vela` 去把网络约束成可部署形式。到了 U85，Arm 又把它和 U55 拉开，补进 transformer operator 支持并给出最高 `4x` 的 ML 性能提升。这说明 Ethos-U 系列内部也不是同一档平台，而是从 MCU 端超低功耗一路延伸到更高性能的 edge AI。

用统一框架看，Ethos-U 的重点不是追求最大绝对吞吐，而是：

- 在严格功耗壳内保持较高推理效率
- 把常见推理算子映射到较固定的硬件路径
- 让系统集成成本和部署路径尽量可控

这类平台的典型判断是：

- `如果你的目标是端侧、低功耗、规则推理`，它很合适
- `如果你追求极高灵活性或大规模训练能力`，它不是主要方向

所以 Ethos-U 代表的是“低功耗专用推理 IP”这一路线。

## 3. 平台型商用 IP 路线：VIP9000 与 Imagination NNA

VeriSilicon VIP9000 和 Imagination NNA 更能代表另一类平台：

- 强调可集成的商业 IP
- 关注多核扩展、数据流压缩、系统协同
- 既讲硬件，也讲软件生态和开发路径

从案例定位看，这类平台通常把预算押在：

- 更完整的商用交付能力
- 更强的可扩展性
- 更明确的软件栈与生态配套

这和纯研究型或超窄场景 IP 的差别在于：

- 不只要“能跑”
- 还要“能卖、能集成、能长期维护”

所以读这类案例时，重点不该放在某个 PE 拓扑细节，而要放在：

- 软件栈成熟度
- 多核和虚拟化能力
- 数据流压缩与工程化能力

它们代表的是“面向量产 SoC 的平台 IP 思路”。

## 4. 端侧整板与完整部署路径：Google Edge TPU

Edge TPU 案例的价值，不在于它一定最通用，而在于它把“从模型到板级部署”这条链做得很具体。

从课件锚点能看出它强调：

- Coral Dev Board / USB Accelerator 这类具体载体
- 板级接口、电源、环境要求
- 模型部署与设备使用链条

Coral 官方文档把平台边界写得非常硬：Edge TPU 只支持 fully `8-bit` quantized 的 TensorFlow Lite 模型，而且模型还必须先用 `edgetpu_compiler` 编译成 Edge TPU 可执行版本。运行时侧不管用 Python 还是 C/C++，都需要 `libedgetpu` 和对应的 TensorFlow Lite 库。这说明它不是“通用 NPU 后端”，而是 `TFLite + compiler + runtime + 设备` 的产品化闭环。

这也解释了它为什么更像“端侧部署工具链”而不是通用编译框架。官方示例里，未被 Edge TPU 吃下的部分可以继续留在 CPU，所以平台价值在于把高收益主路径固化，而不是承诺所有层都能透明上 NPU。

这类平台的关键启发是：

- NPU 不只是一个核，而是完整产品
- 电源、接口、外设和工具链会直接影响可用性
- 边缘部署场景里，“拿到板子能跑起来”本身就是重要价值

所以 Edge TPU 代表的不是“最强架构”，而是“从硬件到开发体验的端侧产品化闭环”。

## 5. 软件栈主导路线：OpenVINO NPU 插件

OpenVINO 案例最值得看的地方不在 NPU 核心本体，而在软件栈组织方式：

- 插件架构
- 模型优化器
- 异步执行引擎
- 与 CPU/GPU/iGPU/NPU 多插件协同

OpenVINO 官方 NPU 文档把这条路线也钉得很实：用户入口是 `compile_model(..., "NPU")`，随后由 plugin / driver / compiler 把 IR 编成设备工件；如果开启 `ov::cache_dir`，后续请求可以直接 import 已缓存模型，而不是每次重新编译。与此同时，当前 NPU 路径仍强调 static-shape 支持，offline blob import 只建议开发用途。也就是说，它确实是“软件栈主导路线”，但不是没有平台边界，而是把这些边界收进统一 API 和插件体系里。

它说明了一件很关键的事：

- 对很多平台来说，用户真正接触到的不是 NPU 寄存器和命令流
- 而是插件、模型转换器、异步推理接口、部署工具链

因此 OpenVINO 的价值押注主要在：

- 部署便利
- 统一软件接口
- 多后端协同
- 模型格式和执行路径打通

这条路线提醒我们：`软件栈成熟度本身就是平台竞争力`

## 6. SoC 异构协同路线：MediaTek APU 与 Qualcomm Hexagon

MediaTek APU 和 Qualcomm Hexagon 更像系统级异构协同案例，而不只是孤立 NPU。

从课程锚点看，它们强调：

- NPU 与 VPU / DSP / 其他单元协同
- SoC 内的内存层次和互联
- 软件栈如何把不同单元组合起来
- 在移动端能效和体验之间做折中

这类平台的典型取舍是：

- 不是所有任务都硬塞给 NPU
- 而是让 DSP、向量单元、张量单元按各自擅长的粒度配合

所以它们代表的不是“纯 NPU 中心主义”，而是：

- `异构系统里，每种单元干最合适的事`

这对于理解真实手机 SoC 特别重要，因为手机 AI 性能经常不是单核 NPU 能力决定的，而是由整个异构执行栈共同决定。

## 7. 案例之间的真正差异，不在名字，而在押注方向

把这些平台放到一起比较，可以看到几种清晰路线：

| 路线 | 代表案例 | 主要押注 |
| --- | --- | --- |
| 低功耗边缘 IP | Arm Ethos-U | 能效、规则推理、可集成性 |
| 商用平台 IP | VIP9000、Imagination NNA | 多核扩展、量产交付、生态能力 |
| 端侧产品闭环 | Edge TPU | 板级部署、开发体验、端到端可用性 |
| 软件栈驱动 | OpenVINO NPU 插件 | 插件体系、统一部署接口、多后端协同 |
| 异构 SoC 协同 | MediaTek APU、Qualcomm Hexagon | 单元协作、移动端能效、系统级分工 |

这张表比参数罗列更重要，因为它直接告诉你：

- 不同平台解决的是不同问题
- 所以它们的架构和软件栈优先级也必然不同

如果再压成三个轴，会更容易形成平台直觉：

| 平台 | 计算核心侧重点 | 软件栈侧重点 | 系统定位 |
| --- | --- | --- | --- |
| `Arm Ethos-U` | 低功耗推理阵列与缓存 | `Cortex-M + Vela` 的嵌入式部署路径 | MCU/endpoint AI IP |
| `VIP9000` | 多核与可扩展数据流 | 商业交付与生态配套 | 量产 SoC 平台 IP |
| `Imagination NNA` | 阵列与控制流协同 | 平台化软件栈 | 商用 IP 路线 |
| `Edge TPU` | 端侧推理硬件 | `TFLite + edgetpu_compiler + libedgetpu` 闭环 | 板级产品路径 |
| `OpenVINO NPU插件` | 核心本体被软件接口抽象掉 | `compile_model / cache / plugin / driver` 统一入口 | 软件栈驱动部署 |
| `MediaTek APU` | NPU/VPU 等单元协同 | SoC 级软件协同 | 移动端异构系统 |
| `Hexagon + NPU` | DSP/向量/张量混合 | 运行时选择与协同 | 移动端异构协同 |

## 8. 怎么用前面八卷的知识拆一个新平台

以后遇到新平台时，不要先查营销页，先按这五步拆：

1. 它主要面向什么场景。
2. 它把核心预算押在阵列、缓存、软件栈还是异构协同。
3. 它的数据流和 memory story 是什么。
4. 它的编译与 runtime 接口是否成熟。
5. 它在性能、功耗、安全、多核、封装里最硬的约束是什么。

如果这五步能回答出来，平台名字换成别的也不影响你理解本质。

同时要避免一种常见误比：

- 不要拿 MCU 边缘 IP 去和手机 SoC 的异构平台比“绝对吞吐”
- 不要拿软件插件栈去和纯硬件 IP 比“核心参数完整度”
- 不要把端侧板级产品和可授权 IP 当成同一类商业对象

比较必须先对齐目标场景，否则结论通常是假的。

## 9. 判断平台案例是否真正读懂的标准

不是会复述产品名，而是能回答：

- 它更像低功耗 IP、商用平台 IP、端侧产品闭环，还是异构协同方案。
- 它为什么要这样组织计算核心和软件栈。
- 它把主要风险放在了哪里：开发生态、功耗、互联、异构调度，还是部署链路。
- 如果换一个目标场景，它为什么可能不再是最优选择。

## 10. 常见误区

- 误区：`厂商案例就是主线知识的替代品`
  - 修正：案例的作用是验证和对照，不是反过来决定主线结构。
- 误区：`看平台只要看算力和精度支持`
  - 修正：软件栈、部署路径、异构协同、功耗壳经常更关键。
- 误区：`同样叫 NPU，平台之间本质差不多`
  - 修正：不同平台押注的系统目标不同，差异可能比名字看上去大得多。
- 误区：`软件插件层不算平台能力`
  - 修正：对真实用户来说，插件、工具链和部署体验往往就是平台能力本身。

这一卷的任务不是把厂商课重新抄一遍，而是把案例重新纳入主线：`平台差异 = 不同系统目标下的不同押注方式`。下一卷会把这些押注重新收敛成一个完整案例，演示从工作负载到硬件、编译、runtime 与系统约束的闭环分析。
