# 01 前置基础

后面的 NPU 架构、数据流、编译器、runtime 看起来像不同层的问题，实际上都绕不开同一组前提：`工作负载的张量形状`、`乘加与归约`、`数据搬运成本`、`并行组织方式`、`性能/能效指标`。这些前提如果没稳住，后面看到 PE、MAC、dataflow、tile、kernel 时就只是在记名词。

## 1. 从神经网络到硬件，先做一次最短翻译

神经网络对硬件并不神秘。把模型压扁以后，硬件真正面对的是三件事：

1. `张量`
   - 一块按 shape 和 layout 排列的数据。
   - 例如输入特征图、权重、输出特征图、本质上都只是不同语义的张量。
2. `算子`
   - 对张量做变换的规则。
   - 例如卷积、矩阵乘、激活、归一化、pooling。
3. `依赖关系`
   - 哪些值必须先算，哪些值可以并行，哪些值可以复用。

硬件设计里最重要的不是“模型名”，而是这几个算子最终退化成什么计算骨架：

- `卷积`：滑动窗口上的乘加与归约
- `GEMM`：规则矩阵乘加
- `Attention`：矩阵乘、缩放、softmax、再乘一次 value
- `MLP`：大矩阵乘 + 非线性

这意味着一大类 AI 负载最后都会被翻译成“很多次乘加 + 若干次非线性 + 大量中间张量搬运”。

在真正进入硬件之前，先把几个最容易混用的词压到同一张表里：

| 词 | 在这里真正指什么 | 常见误解 |
| --- | --- | --- |
| `shape` | 张量每个维度的大小 | 只看总元素数，不看维度顺序 |
| `layout` | 张量在内存里的排布方式，如 `NCHW`、`NHWC` | 觉得只是软件表示，不影响硬件搬运 |
| `operator` | 一种有输入、输出和依赖关系的变换 | 把模型名直接当成硬件操作 |
| `reduction` | 多个乘积或中间值归并到一个结果的过程 | 只看到乘法，不看到累加与部分和 |
| `tile` | 为了片上复用而切出来的一小块计算/数据工作集 | 误以为只是“把大矩阵拆小” |

## 2. 为什么 NPU 讨论里总绕不开矩阵乘加

不是因为 AI 只有乘加，而是因为乘加通常占据了绝大多数算量，而且最适合专用阵列化。

把一个最普通的矩阵乘写出来：

`C[m,n] = Σ_k A[m,k] * B[k,n]`

它天然暴露了 NPU 设计最关心的三个维度：

- `并行度`
  - 不同的 `m`、`n` 可以同时算。
- `归约维`
  - `k` 是不断累加的维度，决定部分和的保存方式。
- `复用机会`
  - `A`、`B`、`C` 三类数据到底谁应该留在本地，谁应该流动。

卷积只是把这个骨架换了一个索引表达。真正做硬件时，经常会把卷积映射为 GEMM，或者至少按 GEMM 的思路分析它的复用和带宽压力。

所以 NPU 不是“会跑神经网络的黑箱”，而是“围绕乘加与归约，尽量减少无效搬运的专用机器”。

## 3. 计算不一定贵，搬运通常更贵

初学 NPU 最容易犯的错，是只盯着算力数量，不盯着数据从哪里来、往哪里去。

原因很简单：

- 一次 MAC 的逻辑不复杂，容易大规模复制。
- 数据一旦离开寄存器文件和局部 SRAM，走到更远层级，单位能量和延迟都会迅速变差。
- 如果同一份权重或激活不能在片上反复复用，再大的阵列也会被带宽饿住。

所以看一台 NPU，先不要问“多少 TOPS”，先问：

1. 权重放哪里。
2. 激活放哪里。
3. 部分和在哪里累积。
4. 什么时候回写。
5. 外存带宽是否足够持续喂满阵列。

如果这五个问题没有答案，峰值算力通常只是宣传值，不是可持续吞吐。

一个很实用的速判法是：

- `阵列忙，但外存读写更忙`：大概率带宽或复用设计不对
- `理论 TOPS 很高，真实层延迟不降`：大概率在等待数据或同步
- `换一个 shape 效率暴跌`：大概率 tile、layout 或边界处理吃亏

## 4. 存储层级决定了大部分真实瓶颈

NPU 讨论中的存储层级通常至少包含下面几层：

| 层级 | 典型角色 | 设计关注点 |
| --- | --- | --- |
| 寄存器 / 寄存器文件 | 保存当前周期最热的数据 | 端口数、访问延迟、面积 |
| PE 本地缓存 / 局部 buffer | 提高单个 PE 或小阵列的复用 | 容量、bank 冲突、数据布局 |
| 共享 SRAM / Global Buffer | 为阵列提供批量数据 | 容量、带宽、仲裁 |
| 片外 DRAM / HBM | 提供总容量 | 带宽、能耗、调度、突发访问 |

真正的性能差距，经常不是出在 MAC 单元上，而是出在：

- tile 切得不对，导致本可复用的数据被重复搬运
- 数据布局和阵列方向不匹配
- buffer 容量不足，部分和反复溢出回写
- DMA、双缓冲、预取没有把搬运和计算重叠起来

后面讲 dataflow、本地 SRAM、DMA、双缓冲时，本质都在回答同一个问题：`如何让最贵的层级尽量少动`。

## 5. 并行不只是一件事

说“这个架构支持并行”太空。硬件里至少要区分四种典型并行方式：

| 并行方式 | 核心思想 | 适合什么 | 代价是什么 |
| --- | --- | --- | --- |
| `SIMD` | 一条指令驱动多个数据元素 | 规则向量操作 | 灵活性一般，控制简单 |
| `SIMT` | 看起来像线程并发，底层仍以束为单位执行 | GPU 风格大规模吞吐 | 分支和访存不规则时效率下降 |
| `MIMD` | 多个核各跑各的指令流 | 控制复杂、任务异构 | 一致性与同步复杂 |
| `Spatial Array` | 把计算映射到空间阵列上，数据在阵列中流动 | 乘加密集、复用明确的负载 | 灵活性较低，需要编译和映射配合 |

NPU 最偏爱的不是“线程并发”，而是`空间并行 + 数据流驱动`：

- 把一类稳定重复的运算映射到很多 PE 上
- 让数据按固定方向流动
- 尽量避免每一步都重新取指、重新做复杂调度

所以 NPU 看上去像“很多小计算单元排成阵列”，不是偶然，而是因为这种组织最容易把控制开销和搬运开销压下去。

## 6. 需要盯住的指标，不只有 TOPS

NPU 设计里最常见的一批指标如下：

| 指标 | 看的是什么 | 常见误读 |
| --- | --- | --- |
| `吞吐` | 单位时间内做了多少有效计算 | 把峰值吞吐当成持续吞吐 |
| `延迟` | 单个请求从进入到完成要多久 | 只看批量任务，不看实时场景 |
| `利用率` | 阵列有多少周期真的在做有效工作 | 只看 MAC 数，不看空泡和等待 |
| `带宽` | 数据供给速度够不够 | 只看峰值带宽，不看访问模式 |
| `算术强度` | 每搬运一份数据能做多少计算 | 不分析复用，无法判断瓶颈在哪 |
| `TOPS/W` | 每瓦完成多少操作 | 忽略精度、稀疏和测试口径差异 |
| `面积效率` | 单位面积产出多少有效能力 | 只堆阵列导致系统其他部分失衡 |

这里最关键的思路是 `Roofline`：

- 当算术强度低时，系统更像`带宽受限`
- 当算术强度高时，系统更像`算力受限`

所以“继续堆 MAC”是不是有意义，必须先看工作负载落在 roofline 的哪一侧。

## 7. 数值格式不是实现细节，而是架构约束

后面专门会讲量化，但在入门阶段至少要先建立一个判断：

- `FP32`：动态范围大，软硬件最省心，成本也最高
- `FP16/BF16`：降低带宽和存储成本，适合很多训练/推理场景
- `INT8/INT4`：推理里最常见的能效优化方向

真正需要盯住的不是“支持哪几种格式”，而是：

1. 乘法位宽是多少。
2. 累加位宽是多少。
3. 中间结果何时截断、饱和或重新量化。
4. 格式切换是否引入额外搬运与控制成本。

很多平台写着“支持 INT4/INT8/FP16”，但如果累加路径、格式转换路径和软件栈没配好，理论优势很难落成真实收益。

## 8. 软件栈不是附属物

只看硬件会误以为 NPU 的工作在阵列里结束了。实际上最小闭环至少包括：

`模型/框架 -> 编译器 -> runtime -> 驱动/固件 -> NPU 硬件`

前置阶段必须先认识这一点：

- 编译器决定算子怎么切、怎么排、怎么映射
- runtime 决定什么时候装载、执行、同步、回收
- 驱动和固件决定命令如何下发、异常如何处理、资源如何隔离

后面看到任何“硬件特性”，都应该追问一句：`软件栈有没有能力把它稳定用起来`

## 9. 学完本章后，应该形成的最小判断能力

至少能独立回答下面这些问题：

- 为什么神经网络工作负载天然适合空间并行和专用数据流。
- 为什么数据复用和存储层级经常比“再加几个 MAC”更重要。
- 为什么并行模型不同，会直接改变控制方式和能效上限。
- 为什么 TOPS、带宽、利用率、TOPS/W 必须一起看。
- 为什么编译器、runtime、驱动从第一天起就是 NPU 设计的一部分。

## 10. 常见误区

- 误区：`NPU = 只会做矩阵乘的 GPU`
  - 修正：GPU 仍然保留强线程模型和较高通用性，NPU 往往更激进地围绕固定数据路径做优化。
- 误区：`算力越大，系统一定越快`
  - 修正：带宽、容量、同步、映射效率都可能先成为瓶颈。
- 误区：`低精度只是为了省一点存储`
  - 修正：低精度同时影响带宽、阵列宽度、累加路径和能效。
- 误区：`硬件做好以后，软件自然能适配`
  - 修正：很多架构收益只有在编译器与 runtime 能把工作负载喂成“硬件喜欢的形状”时才成立。

这一章的作用不是讲完 NPU，而是把后面所有章节都要反复使用的坐标轴一次性立住。

下一卷不再解释这些前提本身，而是直接回答：在这些前提都成立的情况下，为什么系统会自然走向 NPU，而不是停在 CPU/GPU/FPGA 的原有分工上。
