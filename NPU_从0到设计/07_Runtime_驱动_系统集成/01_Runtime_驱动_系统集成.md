# 07 Runtime、驱动、系统集成

前一卷讲的是“计划怎么生成”，这一卷讲“计划怎么执行”。编译器把模型变成可执行工件以后，真正让系统跑起来的，是 runtime、驱动、固件、内存管理和同步机制共同组成的执行链。

如果编译器回答的是：

- 哪些算子上 NPU
- tile 怎么切
- buffer 怎么分
- codegen 生成什么工件

那么 runtime 和驱动回答的就是：

- 这些工件怎么装载
- 任务怎么提交
- 内存怎么映射
- 执行何时完成
- 出错时谁负责恢复

## 1. 先把分层讲清楚：谁负责什么

一个典型 NPU 软件执行栈可以压成下面几层：

| 层 | 主要职责 |
| --- | --- |
| `应用 / 推理框架` | 调用模型推理接口、组织业务流程 |
| `Runtime` | 装载模型、创建上下文、管理 stream/queue、调度执行、同步与回调 |
| `HAL / 用户态库` | 屏蔽驱动细节，提供统一 API |
| `内核驱动` | 设备发现、资源管理、中断、内存映射、命令提交、IOCTL/mmap |
| `固件 / 硬件调度器` | 解析命令、驱动硬件执行、上报状态 |
| `NPU 硬件` | 真正完成计算、DMA、同步、异常产生 |

分层的意义不在“画架构图好看”，而在于把责任切开：

- runtime 决定执行策略
- 驱动决定系统资源如何安全进入硬件
- 固件和硬件决定命令如何真正落地

只要边界一乱，问题就会变成：谁都能改一点，但没人能解释端到端为什么慢或为什么不稳。

## 2. Runtime 的核心任务：把静态工件变成一次可控执行

runtime 至少要负责五件事：

1. `初始化与资源获取`
   - 设备、context、stream、模型句柄、工作缓冲区。
2. `模型装载`
   - 把编译器产物读入并准备可执行状态。
3. `输入输出组织`
   - tensor 描述、数据布局、host/device 缓冲管理。
4. `任务提交与同步`
   - 同步执行、异步执行、回调、事件、完成通知。
5. `错误与生命周期管理`
   - 超时、异常、资源释放、句柄失效处理。

所以 runtime 不是“包一层 API”，而是把编译器产物变成`可管理、可重复、可并发、可恢复`的执行实例。

## 3. 驱动的核心任务：把系统资源安全地交给硬件

驱动层真正要处理的是内核和设备边界问题：

- 设备探测与初始化
- 寄存器映射
- 中断注册
- 命令缓冲区与任务提交通道
- 内存管理与地址映射
- 用户态接口（如 IOCTL / mmap）

它的关键目标不是“多写一点硬件细节”，而是：

- 让硬件对上层呈现稳定接口
- 让资源在多进程、多任务下仍然可控
- 让错误不会直接把系统拖垮

所以驱动真正难的地方往往不在“会不会访问寄存器”，而在：

- 并发
- 生命周期
- 中断与同步
- 内存一致性

## 4. 模型与工件装载：文件不是直接喂给硬件的

无论是 `.om` 这类模型包、OpenVINO 编译产物，还是自定义后端生成的 binary，真正进入执行态前通常都要经历：

1. 读取工件元数据
2. 分配或绑定执行所需缓冲区
3. 解析 kernel / command / descriptor
4. 建立 runtime 句柄与资源关系

这说明“模型加载”不是单纯文件 IO，而是把静态工件转为：

- 可访问的指令或命令缓冲
- 已绑定的 tensor 视图
- 可提交的执行实例

所以一旦模型格式、内存布局、runtime 版本不一致，问题往往会在装载阶段就暴露，而不是等到真正执行时才发现。

拿 OpenVINO NPU 官方路径做例子，这条链不会停在“把 IR 读进来”。用户是通过 `core.compile_model(model, "NPU")` 让 NPU plugin 把模型编译成设备可执行表示；如果开启 `ov::cache_dir`，第一次编译后的工件会进入 OpenVINO cache，后续 cache hit 时插件直接 import model，而不是重新 compile。也就是说，runtime 面对的通常不是原始训练模型，而是“已编译模型 / 可导入工件”的生命周期管理。

官方文档还明确写了两个容易被忽视的边界：当前 NPU 路径强调 static shape 支持；offline compilation 和 blob import 只建议开发用途，跨 OpenVINO 版本的 blob 兼容性并不保证。缓存能降低首轮启动延迟，但它不会自动抹掉平台版本边界。

## 5. 内存管理：系统集成里最容易出事的一层

NPU 运行时至少要同时处理三类内存问题：

- `Host 内存`
  - 应用和 runtime 能直接访问
- `Device 内存`
  - NPU 真正执行时使用
- `Host <-> Device 传输`
  - H2D / D2H、zero-copy、映射、cache 一致性

在 Linux 系统里，经常还会遇到：

- CMA
- ION 或等价分配器
- DMA mapping API
- 用户态 mmap

这层如果没处理好，最典型的问题就是：

- 复制次数太多
- cache 不一致
- 对齐和粒度不匹配
- 大块连续内存难分配

所以运行时和驱动里，内存管理不是配角，它常常直接决定端到端延迟和稳定性。

一个很实用的系统问题是：`一次推理到底发生了几次复制`。如果这个问题回答不清，runtime 层的很多“优化”都可能只是把复制位置换了一下，并没有真正把延迟压下去。

再往系统外看一步，OpenVINO 的 NPU 配置文档也提醒了一个现实约束：NPU 能不能用，不只取决于 API，还取决于 host driver 和 OS 前提。Windows 侧通常依赖系统驱动分发或手动安装 NPU driver；Ubuntu 侧则明确给出了版本和内核下限。把这些前提漏掉，运行时问题就很容易被误判成“模型或算子问题”。

## 6. 命令缓冲区与任务提交：真正的“执行入口”

编译器生成的计划，最终往往会转成：

- command buffer
- descriptor chain
- kernel binary + 参数表
- queue entry

驱动/runtime 的任务是把这些内容按硬件要求提交出去。这个过程通常包含：

1. 填写命令缓冲区
2. 绑定输入/输出地址
3. 配置同步对象或完成通知
4. 提交给硬件调度器
5. 等待完成、中断或回调

这一步为什么关键：

- 它决定 CPU 和 NPU 是否频繁交互
- 它决定同步点粒度是否过细
- 它决定多 stream 或多任务场景能否保持吞吐

所以“命令提交流程”本身就是性能设计对象，不是机械搬运。

## 7. 中断、回调、同步：系统吞吐和可用性的分界线

NPU 执行不是调用一个函数然后世界静止等待。真正系统里必须处理：

- 任务完成中断
- 错误中断
- DMA 完成事件
- 异步提交与回调
- 同步等待与超时

设计这层时，要特别避免两种极端：

- `全同步`
  - 简单，但 CPU 与 NPU 并行性差
- `过度异步`
  - 吞吐可能好看，但调试、错误恢复和资源回收会变难

一个成熟执行栈通常会提供：

- 同步模式，方便简单场景和调试
- 异步模式，提升吞吐和多任务重叠
- 明确事件与回调接口，帮助上层组织 pipeline

把两种执行方式压成对比，更容易看清取舍：

| 模式 | 优点 | 风险 | 适合什么阶段 |
| --- | --- | --- | --- |
| `同步执行` | 语义简单、易调试、问题边界清楚 | CPU/NPU 重叠少，吞吐不高 | bring-up、单请求低复杂度场景 |
| `异步执行` | 更容易隐藏等待、提高设备利用率 | 生命周期、回调、同步和错误处理更复杂 | 稳定量产路径、高吞吐场景 |

## 8. Context、Stream、Queue：不是术语，而是资源隔离模型

很多 runtime 都有类似概念：

- `Context`
  - 一组和设备、内存、状态相关的执行环境
- `Stream`
  - 一条相对独立的执行流
- `Queue`
  - 待提交任务的有序容器

它们存在的意义：

- 把资源生命周期说清楚
- 让并发执行有明确归属
- 让同步作用域可控

没有这些抽象，异步执行、多模型并行、动态 batch、多进程共享设备都会迅速失控。

## 9. 为什么异步推理通常是系统吞吐的关键

只做同步执行，流水线通常会变成：

`拷输入 -> 提交 -> 等待 -> 取输出 -> 再拷下一份`

异步执行的价值是把这些阶段重叠起来：

- 当前任务计算时，下一任务准备输入
- 当前任务等待回调时，CPU 可以组织其他工作
- 多个 stream 可以覆盖部分设备与总线空闲

但异步并不是越多越好。需要同时控制：

- 并发 stream 数量
- 内存占用
- 回调复杂度
- 顺序依赖

如果 runtime 只是一味增加异步请求，却没有配套的内存和同步设计，吞吐很可能上不去，系统复杂度却陡增。

## 10. 驱动 bring-up：真正让设备“活起来”的路径

从系统集成角度看，一个 NPU 驱动通常要经历这条 bring-up 路径：

1. 设备树或平台资源描述就位
2. probe 成功
3. 寄存器映射和基础初始化正确
4. 中断能正常注册并响应该设备
5. DMA / 内存路径可用
6. 命令提交通路可用
7. 最小任务闭环跑通
8. 再逐步打开异步、并发、多流、错误恢复

这条顺序为什么重要：

- 最小闭环没稳时，先追异步和高吞吐只会让定位更难
- probe、内存映射、中断、命令提交通常是一条串联链，前面任何一点不稳，后面都不成立

实际定位系统问题时，也可以先按症状分层：

| 症状 | 优先怀疑哪层 |
| --- | --- |
| 模型加载就失败 | 工件格式 / runtime 初始化 / 版本边界 |
| 任务提交后一直无响应 | 命令提交链 / 中断 / 调度器握手 |
| 吞吐明显上不去 | 内存复制、同步粒度、异步深度、stream 组织 |
| 长跑后不稳定 | 资源生命周期、回收、错误恢复、并发边界 |

## 11. 系统集成视角下，最关键的不是“API 会不会调”，而是边界会不会失控

真正让 NPU 系统变脆弱的，往往不是某个 API 调错，而是边界没有被明确建模：

- 资源归属不清
- 输入输出缓冲生命周期不清
- 错误谁来回收不清
- 异步回调后谁负责同步不清
- 编译器工件和 runtime 版本约束不清

OpenVINO 2026 还把 `compiler_type` 暴露成显式属性，允许 plugin 在 driver 内置 compiler 与 OpenVINO 包内 compiler 之间切换。这个细节说明 runtime 行为从来不是单层软件决定的，而是 `driver + plugin + compiler library` 的联合作用。只盯着 API 表层，很容易漏掉真正影响稳定性和首轮时延的那一层。

所以系统集成层必须把下面几件事写成稳定约定：

- ABI / 工件格式
- 内存所有权
- 同步模型
- 错误码与恢复路径
- profile / log / debug 通道

这些边界从编译阶段怎样形成，可回看 [06_编译器_图优化_后端生成/01_编译器_图优化_后端生成.md](../06_编译器_图优化_后端生成/01_编译器_图优化_后端生成.md)。

这五件事不稳，系统跑得再快也很难长期维护。

## 12. 判断这一卷设计是否过关的五个问题

1. `分层职责是否清楚`
   - runtime、HAL、驱动、固件、硬件各负责什么。
2. `命令提交流程是否完整闭环`
   - 工件装载、缓冲绑定、提交、完成通知、回收是否说清楚。
3. `内存管理是否与性能目标一致`
   - 是否控制了复制次数、对齐、映射和一致性。
4. `同步与异步模型是否可控`
   - 是否既能调试，又能在高吞吐场景下重叠执行。
5. `系统 bring-up 顺序是否合理`
   - 是否先跑通最小闭环，再逐步放开复杂能力。

## 13. 常见误区

- 误区：`runtime 就是驱动上面那层薄封装`
  - 修正：runtime 实际负责上下文、资源、同步、回调和执行策略，是系统行为的主要组织者。
- 误区：`驱动只要能收发命令就行`
  - 修正：中断、内存映射、并发、错误恢复和资源隔离才是长期稳定的关键。
- 误区：`异步一定比同步好`
  - 修正：没有明确资源和同步模型的异步，只会放大复杂度与调试难度。
- 误区：`模型能加载成功，就说明系统集成差不多了`
  - 修正：真正困难的部分通常在长期运行、多 stream、异常处理和内存压力场景。
- 误区：`系统集成问题主要是软件细节`
  - 修正：它们往往直接暴露硬件接口、DMA 路径、同步机制和命令格式设计是否合理。

这一卷建立的是第四条系统因果链：`编译器工件 -> runtime 资源组织 -> 驱动与内存映射 -> 命令提交 -> 中断/回调/同步 -> 真实系统执行行为`。下一卷会继续站在系统层往上看：性能、功耗、安全、多核和 Chiplet 怎样反过来限制架构与执行策略。
