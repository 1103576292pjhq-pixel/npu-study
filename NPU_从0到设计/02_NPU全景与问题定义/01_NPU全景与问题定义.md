# 02 NPU全景与问题定义

如果没有先把“为什么需要 NPU”讲清楚，后面所有微架构细节都会变成散点。因为 PE、脉动阵列、量化、编译器后端、runtime 都不是孤立优化，它们共同服务于同一个目标：`在给定功耗、面积、成本、延迟约束下，把 AI 负载更高效地跑起来`。

## 1. NPU 不是为了“跑 AI”而生，而是为了在约束下跑 AI

把问题说完整，NPU 要解决的不是“能不能算”，而是：

- 同样的 AI 工作负载，能不能在更低功耗下完成
- 同样的功耗和面积预算，能不能给出更高吞吐
- 同样的系统约束，能不能把数据搬运压到更低
- 同样的模型，能不能用更可控的方式部署到边缘端、手机、汽车、数据中心

这就是 NPU 和“拿一个通用处理器硬跑模型”的根本区别。

CPU、GPU、FPGA 都能做 AI 计算，但它们的默认优化目标并不一样：

- CPU 优先保证广泛软件都能高效运行
- GPU 优先保证大规模线程并行吞吐和编程生态
- FPGA 优先保证可重构和定制化
- NPU 优先保证神经网络算子的`有效吞吐/能效/部署成本`

所以 NPU 从第一天起就是一个带约束的专用解，而不是“更强的一般处理器”。

把四类处理器放在一张表里看，会更容易抓住 NPU 的问题定义：

| 类型 | 默认优化目标 | 擅长什么 | 为什么不是 NPU 的直接替代 |
| --- | --- | --- | --- |
| `CPU` | 通用性、低延迟控制 | OS、框架、控制流、零碎任务 | 规则乘加的能效不高 |
| `GPU` | 大规模线程吞吐 | 训练、通用并行计算、张量运算 | 灵活性本身要付控制和功耗成本 |
| `FPGA` | 可重构和快速定制 | 验证新数据流、特殊算子 | 面积/功耗效率与生态通常不如定制 IP/ASIC |
| `NPU` | 约束下的有效 AI 吞吐 | 推理主路径、固定模式加速 | 通用性更低，强依赖软件栈配合 |

## 2. AI 工作负载到底给硬件提出了什么要求

神经网络负载最典型的特征是：

1. `规则的大规模乘加`
   - 卷积、GEMM、MLP、attention 中的大头都能归约为乘加。
2. `中间张量巨大`
   - 算完不难，存下、搬动、复用才难。
3. `数据复用机会明确`
   - 权重、激活、部分和都存在重复使用的可能，只是复用方向不同。
4. `低精度可接受`
   - 很多推理任务用 FP16、INT8、甚至更低位宽仍能保持可用精度。
5. `图级优化空间大`
   - 算子融合、layout 变换、内存复用、静态调度都可能显著改变效率。

这五个特征联合起来，几乎直接指向了 NPU 的设计方向：

- 专用阵列而不是纯线程化
- 强调片上数据复用而不是频繁回外存
- 明确支持低精度
- 强依赖编译器和 runtime 做全局规划

## 3. 为什么 CPU 不适合当主力 NPU

CPU 的优势从来不在“海量规则乘加”，而在：

- 复杂控制流
- 丰富缓存体系
- 高单线程性能
- 强通用性

这使它非常适合：

- 操作系统
- 框架调度
- 运行时控制
- 零碎、不规则、分支密集型任务

但把 CPU 直接拿来跑大规模神经网络时，问题会很明显：

- 标量或小向量单元难以和大规模乘加需求匹配
- 指令取用、分支预测、乱序执行等机制对规则乘加不是最省能的路径
- 缓存对 AI 张量的复用方向不一定友好

所以 CPU 在 NPU 系统里通常更像：

- 系统主控
- runtime 承载者
- NPU 命令与内存管理的协调者

而不是主要算力来源。

## 4. GPU 很强，但它优化的是另一种“高吞吐”

GPU 比 CPU 更接近 AI 计算需求，因为它天然擅长大规模并行。但 GPU 仍然不是所有场景下最优的 NPU 替代物。

GPU 的强项：

- 大规模线程并发
- 高吞吐浮点与张量运算
- 成熟的软件生态
- 训练和通用并行计算都能覆盖

GPU 的局限：

- 线程调度、寄存器管理、缓存层级仍然服务于更通用的执行模型
- 对边缘端、低功耗端来说，能效和成本不一定合适
- 若工作负载已知且规则，GPU 的灵活性本身就是一部分额外成本

一句话概括：

- `CPU` 像系统指挥部
- `GPU` 像大规模并行工厂
- `FPGA` 像可重构实验平台
- `NPU` 像围绕 AI 主负载做过深度工艺化裁剪的专用产线

所以很多边缘端或 SoC 场景里，系统不会问“GPU 能不能算”，而会问：

- 在 1W、2W、5W 这种功耗壳里，GPU 算得是否划算
- 在手机/摄像头/车载 SoC 里，GPU 是否值得为某个固定 AI 负载一直保留这么高的通用性

这恰恰是 NPU 的切入点。

## 5. FPGA 的角色：可重构，但不天然适合作为统一结论

FPGA 很适合：

- 快速验证新数据流
- 尝试特殊位宽和特殊算子
- 在需求变化大时保留灵活性

但 FPGA 不是 NPU 设计讨论的默认终点，因为：

- 面积效率通常不如 ASIC/IP
- 功耗效率通常也不如定制化阵列
- 部署和软件生态复杂度更高

所以在“从零理解 NPU 设计”这条主线上，FPGA 更像过渡平台或研究平台，而不是最终价值函数本身。

## 6. NPU 真正的价值函数

判断一套 NPU 设计值不值，至少要同时看下面六个维度：

| 维度 | 核心问题 |
| --- | --- |
| `性能` | 是否能在目标模型和目标 batch 下给出足够吞吐与延迟 |
| `能效` | 每瓦能做多少有效工作，热和续航是否可接受 |
| `带宽压力` | 外存是否成为主要瓶颈，片上复用是否足够 |
| `面积与成本` | 阵列、SRAM、NoC、控制器、软件栈值不值得这部分硅预算 |
| `可编程性` | 编译器与 runtime 是否能把真实模型稳定映射上去 |
| `可集成性` | 是否容易接入 SoC、驱动、框架、部署流程 |

这也是为什么任何只谈“TOPS”的 NPU 介绍都不完整。NPU 从来不是单指标问题，而是一个多约束折中问题。

## 7. NPU 在系统里的真实位置

把一条完整路径写出来，通常是：

`模型 -> 框架 -> 编译器 -> runtime -> 驱动/固件 -> NPU 硬件 -> 内存/总线/外设`

这条链说明两件事：

1. NPU 不是孤立芯片
   - 它一定嵌在系统里，与 CPU、内存、总线、OS、驱动协同。
2. NPU 不是只靠硬件决定结果
   - 编译器和 runtime 决定了阵列究竟有没有被喂饱。

所以后面所有章节都要带着系统视角读：

- 一个好看的数据流，如果编译器很难映射，就未必是好架构。
- 一个很强的阵列，如果 runtime 需要高同步成本，也可能不划算。
- 一个很高的峰值吞吐，如果内存系统跟不上，就只是纸面能力。

## 8. 什么样的问题定义，才能导出靠谱的 NPU 设计

真正做 NPU，不是先问“我想做一个多少 TOPS 的芯片”，而是先把问题定义清楚：

1. 目标负载是什么
   - CNN、Transformer、混合模型，还是特定行业模型。
2. 部署场景是什么
   - 手机、边缘盒子、车载、云端卡。
3. 延迟/吞吐目标是什么
   - 单帧实时、批处理吞吐，还是离线高并发。
4. 功耗和面积预算是多少
   - 这会直接决定阵列、SRAM 和接口能力。
5. 允许的数值格式是什么
   - FP16、BF16、INT8、INT4，不同格式改变整个数据路径。
6. 软件栈准备到什么程度
   - 是否已有编译器后端、runtime、驱动、模型转换流程。

如果这些条件没定义清楚，谈架构优劣基本都在漂。

## 9. 看待 NPU 的正确方式：先看瓶颈，再看结构

面对任何一个 NPU 平台，推荐按这个顺序理解：

1. 目标工作负载是什么。
2. 当前最贵的是算力、带宽、容量还是功耗。
3. 架构通过什么方式缓解这个最贵项。
4. 为此牺牲了哪些灵活性或软件复杂度。
5. 这些牺牲是否符合目标场景。

举例：

- 边缘端 NPU 往往把预算押在低功耗、低精度、固定功能块和轻量 runtime 上。
- 云端或高性能平台会更重视可扩展性、编程模型、内存系统与多核协同。

结构不是孤立选择，它总是对着某个瓶颈和某个场景作出的。

所以真正做问题定义时，最怕的不是“信息不够多”，而是跳过了这一步，直接谈阵列或 TOPS。

## 10. 本卷之后应该得到的判断能力

至少能回答：

- 为什么 AI 负载会逼出 NPU 这种专用架构，而不是继续只靠 CPU/GPU。
- 为什么 NPU 的核心价值在“有效吞吐/能效/部署成本”，不是单一峰值算力。
- 为什么 NPU 从定义上就离不开编译器、runtime 和系统集成。
- 为什么任何架构讨论都必须带着工作负载和场景一起看。

## 11. 常见误区

- 误区：`NPU 就是 AI 版 GPU`
  - 修正：两者都能做张量计算，但默认优化目标、控制方式、功耗预算和系统位置并不一样。
- 误区：`有了 NPU，CPU/GPU 就不重要了`
  - 修正：CPU 负责系统控制，GPU 在许多训练和通用并行场景仍然关键，NPU 只是重新分工，不是替代一切。
- 误区：`NPU 只和硬件工程师有关`
  - 修正：没有编译器、runtime、驱动和框架接入，NPU 的大部分价值无法兑现。
- 误区：`只要模型能跑起来，架构就算成功`
  - 修正：真正的成功标准是目标场景下的性能、功耗、成本、软件可维护性一起过关。
