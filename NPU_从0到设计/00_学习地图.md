# 00 学习地图

这套笔记的目标不是把 20 门课按原顺序重排，而是建立一条能从零走到 NPU 设计的稳定问题链：

`工作负载长什么样 -> 为什么通用处理器不合适 -> 计算单元如何组织 -> 数据如何复用 -> 位宽如何压缩 -> 编译器如何映射 -> runtime 如何落地 -> 系统约束如何反压架构 -> 最终怎样形成可验证闭环`

## 先抓住的总图

看 NPU，永远先看五件事：

1. `算什么`
   - 是卷积、GEMM、attention，还是一串图算子。
2. `数据怎么走`
   - 权重、激活、部分和分别停在哪里，谁在移动，谁在复用。
3. `哪里最贵`
   - 算力、片上 SRAM、外部带宽、功耗、面积、软件复杂度，真正限制的是哪一个。
4. `谁来决定执行顺序`
   - 是硬件控制器、编译器调度、runtime 命令队列，还是几者共同决定。
5. `为了什么牺牲了什么`
   - 更高能效通常意味着更低灵活性；更强可编程性通常意味着控制和数据搬运开销更大。

如果一段材料不能回答这五个问题，它就还没有真正解释 NPU。

## 三层心智模型

### 第一层：算子层

先把神经网络看成张量变换，不急着想“模型有多先进”。

- 卷积、本质上是局部窗口复用下的乘加累积
- GEMM、本质上是规则数据访问下的大规模乘加
- Attention、本质上是更复杂的数据依赖与带宽压力
- 激活、归一化、pooling 等通常不是算力瓶颈，但经常是融合与访存瓶颈

### 第二层：微架构层

所有 NPU 都在回答同一组问题：

- 乘加单元怎么堆
- 部分和在哪里累积
- 权重和激活谁复用、谁流动
- 控制粒度到指令、tile、算子还是图
- 片上存储怎么分层

这层如果没看清，就会误把“TOPS 高”当成“系统一定快”。

### 第三层：系统层

真正落地时，性能不只由 MAC 数量决定，还取决于：

- 编译器能不能把图切成硬件喜欢的形状
- runtime 能不能把执行顺序和同步开销压住
- 驱动和系统总线会不会让数据搬运拖垮阵列
- 热、功耗、安全、多核互联是否允许设计长期稳定运行

## 一条贯穿全书的判断链

读每一章时，强制自己按下面顺序判断：

1. 目标工作负载的张量形状与依赖关系是什么。
2. 理论计算量和理论访存量分别是多少。
3. 哪种数据流能最大化权重/激活/部分和复用。
4. 片上 buffer、register file、DMA 是否支持这种复用。
5. 需要什么精度、量化、稀疏或融合来降低成本。
6. 编译器如何把算子图切成 tile、schedule、kernel。
7. runtime 如何下发、同步、复用内存并处理异常。
8. 最终瓶颈是算力、带宽、容量、功耗，还是软件栈。

这八步反复出现，只是粒度在变。

## 最容易走偏的地方

- 把 NPU 理解成“专门跑 AI 的 CPU”
  - 错在没有看到数据流、片上存储与专用数据通路才是核心。
- 只看 TOPS，不看 TOPS/W、带宽和可用率
  - 峰值算力往往比可持续吞吐乐观得多。
- 把量化、稀疏、融合看成纯算法技巧
  - 它们真正有价值，是因为能改变数据搬运和存储压力。
- 把编译器和 runtime 当成“软件附属品”
  - 没有编译和运行时，硬件的可用率、可编程性和部署路径都不成立。
- 被厂商命名带偏
  - 各家名字不同，底层仍然要回到 PE、buffer、dataflow、schedule、runtime 这些通用抽象。

## 学到什么程度算真正过关

不是记住术语，而是能自己做下面这些判断：

- 给定一个卷积层或 GEMM，能说清哪部分最可能卡带宽。
- 看到一个阵列结构图，能判断它更像 weight stationary、output stationary 还是别的复用方向。
- 看到一个“支持 INT8/INT4/FP16”的宣传语，能追问数据路径、累加位宽和转换开销。
- 看到编译器后端优化，能判断它是在省算、省搬运，还是省同步。
- 看到一个平台案例，能说清它把预算押在算力、存储、软件栈还是系统集成上。

## 如果想最快起步，先抓三卷

1. 先读 `01_前置基础`，把后续各卷反复回指的张量、算子、并行、访存和能效坐标轴稳住。
2. 再读 `02_NPU全景与问题定义`，把“为什么是 NPU、不是别的”这件事钉牢。
3. 再读 `03_计算核心与阵列架构`，让 PE / MAC / 阵列 / dataflow / runtime 这些词落到可判断的硬件对象上。

- 读者已经知道后面每一卷为什么存在。

- 可以从“模型工作负载”一路追到“硬件与系统瓶颈”。
